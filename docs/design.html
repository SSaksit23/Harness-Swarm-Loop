ARBOR · application design graph-based agent loop harness workbench spec v0.3 · editable canvas + memory store + telemetry

One root. Three branches. Every run gated by something that can say no.

ARBOR is a tree-based workbench for engineering controlled autonomous dev systems. The user plants a Mission Root — goals, context, and metric scope entered as input labels — and the tree wires that root through the three HSL layers: the Harness (what the agent knows and must obey), the Swarm (who does the work), and the Loop (when it runs and when it stops).

root harness — the standard swarm — the workers loop — the clock & the stop gate / halt
01 · system tree — live editor

The tree is the application

Everything in ARBOR is a node on one tree, and this canvas is a working editor, not a picture. Select a node to inspect its role and contract — or build your own: add nodes and sub-nodes from the toolbar, drag any node to reposition it, and drag the small ○ port on a node's right edge onto another node to wire a new connection. Click an edge to select it; Delete removes a selected node or edge. Your layout is saved in this browser; Reset restores the canonical HSL tree.

click = inspect · drag node = move · drag ○ port → node = connect · sub-node attaches under the selected node
02 · input labels

What the user plants at the root

A run begins with one form — filled by hand, or drafted automatically by the label compiler below from a single plain-language description. Either way, each field is a labeled input on the Mission Root, and each label feeds a specific branch — nothing is decoration; every label becomes a constraint somewhere in the tree.

goalrequired

What "done" means, stated as an outcome the verifier can check — not a vibe. One sentence of intent plus explicit success criteria.

→ harness/brief · success_criteria[]
contextrequired

Repo paths, architecture notes, rules, and out-of-scope declarations. Merged with recalled memory at session start to form the brief.

→ harness/brief · harness/memory
metric_scoperequired

The measurable envelope: which metrics count (tests passing, lint clean, eval score ≥ threshold) and over what surface (files, packages, endpoints).

→ swarm/verifier · rubric + scope
budgetdefault: capped

Max iterations, cost ceiling in tokens or dollars, and the no-progress window. Omitted values fall back to conservative defaults — never to unlimited.

→ loop/contract · loop/hard_stops
triggerdefault: manual

What starts a tick: manual run, schedule (cron), or event (CI failure, new backlog item). Scope of each tick is bounded by the contract.

→ loop/contract · trigger + scope
width_hintdefault: auto

Feeds the ceiling test: is this task genuinely wide (parallel workers pay off) or sequential (one agent, no swarm)? Auto runs the test; the user can override.

→ swarm/orchestrator · topology decision

Auto-generated labels — the label compiler

Nobody has to fill six fields. The user can describe the mission in one plain-language box, and the label compiler — a single premium-model pass — drafts every label from it, quoting the span of the user's text each value came from. Labels below the confidence threshold are flagged, and the run stays blocked until the user confirms or edits them: the LLM drafts, the human ratifies, and invariant 1 still holds because an unconfirmed goal never compiles to success criteria.

freeform input→ LLM extracts labels + confidence + source span→ confidence < 0.8 → flag→ user confirms / edits→ labels planted on root

03 · data model

Everything is a node; every handoff is a contract

The tree persists as plain JSON on disk inside the project (versionable, diffable, no database). Two schemas carry the whole system: the Node, and the Edge contract that types every handoff between nodes — orchestrator to worker, worker to verifier, verifier to loop. 04 · project memory store

Every project remembered, every memory queryable

Files stay the source of truth for a single project — diffable, git-versionable, reviewable. The database sits behind them as the memory store: it indexes every project's tree, ticks, and crystallized memory so recall works across projects ("have we solved this before, anywhere?") and survives repo deletion. The store is a projection of the files — rebuildable at any time, so losing it never loses work.

tier 1 · embedded

SQLite + sqlite-vecrecommended default

One file (arbor/index.db), zero setup, transactional. FTS5 handles keyword recall; sqlite-vec handles semantic recall over memory embeddings. Right up until you need multiple users or machines.

fits: solo dev · one machine · CI-friendly
tier 2 · document db

MongoDB Atlas

The node / tick / memory records are already documents, so they map 1:1 into collections — no translation layer. Atlas Vector Search covers semantic memory recall, and one cluster serves every project and teammate.

fits: team · many projects · hosted dashboards
tier 3 · relational

Postgres + pgvector

Choose it when the memory store must join with existing company data or Postgres is already running. JSONB columns keep the document shape; pgvector covers recall.

fits: existing pg infra · compliance needs

05 · run cycle

What one tick does

Each loop tick walks the tree root-to-leaves and back. The column of color shows which layer owns each stage — the same three hues as the tree, with red reserved for the moments the system can refuse.

T1harnessLoad context. Read the brief, recall memory, mount relevant skills. The success criteria loaded here are the standard everything downstream is judged against.
T2swarmPlan & split. The orchestrator (premium model) runs the ceiling test, then decomposes the tick's scope into typed tasks. Small or sequential work skips the swarm entirely.
T3swarmExecute in parallel. Specialist workers (cheaper models) each take one task inside the sandbox and return the typed payload — diff, claims, artifacts.
T4gateVerify. The verifier checks every payload against the harness's success criteria and the metric scope. It can reject: failed items are requeued or the tick fails. Claims are never trusted — only checks.
T5loopDecide. Continue if the verifier failed and budget remains; stop on verifier pass, max iterations, no-progress window, or cost ceiling — whichever fires first.
T6harnessCrystallize. Write the tick record, distill what was learned into memory, and promote repeated work into skills. The next tick starts smarter than this one.

08 · architecture

Three thin layers over the filesystem

Workbench (frontend). React + SVG tree canvas (the editor above is the seed of it), inspector forms generated from the node schemas, and a websocket feed of tick events. No state of its own — it renders the tree files and the event stream.

Engine (runner). A Claude Agent SDK process per run: it loads the harness, spawns orchestrator / worker / verifier agents per the swarm config (model-tiered — the premium model only plans and verifies designs; workers run on cheaper models), enforces the loop contract, and emits tick records. Hard stops are enforced by the engine, outside any model's reach — a model cannot talk its way past the cost ceiling.

Store (filesystem). An arbor/ directory in the project: tree/*.json for nodes and contracts, ticks/*.json for run history, memory/ and skills/ for the compounding layer. Everything is git-versionable, so the control system itself has a reviewable history. Behind the files sits the memory store from §04 — SQLite by default, MongoDB Atlas at team scale — indexing every project for cross-project recall.

Plugin mapping. Each branch delegates to the layer plugins already built: harness → agent-md-authoring + compounding-dev-stack + skill-curator; swarm → agent-swarms + orchestrator-delegation; loop → loop-engineering + self-verifying-loop. ARBOR is the wiring diagram; the plugins are the components.

09 · control invariants

Four rules the tree may never violate

The workbench validates these on every save and refuses to start a run that breaks one. This is what keeps the system control rather than chaos. Free wiring on the canvas is allowed while you sketch — the validator runs at save-and-run time, so an experimental edge costs nothing until you ask the engine to execute it.

invariant 1

A standard exists

The root's goal label must compile to checkable success criteria in the brief. No run starts against a vibe.

drop it → confident garbage ships
invariant 2

Something can say no

Every path from workers to output passes through a verifier node wired to the standard. Claims are evidence of nothing.

drop it → no one checks the work
invariant 3

Every loop halts

All three hard stops — max iterations, no-progress window, cost ceiling — must be finite, and the engine enforces them outside the model.

drop it → the budget burns at 3am
invariant 4

Every run teaches

A tick is not complete until its record and lessons are crystallized back into memory. Failures instruct instead of repeating.

drop it → every run starts cold
